# KARPATHY-INTEGRATION.md

## 目的
在不改写 vendor 原文的前提下，将 `~/.codex/vendor_imports/andrej-karpathy-skills/CLAUDE.md` 作为代码任务的详细原则来源，并明确它与现有全局规则的作用域、优先级与落地方式。

## 原则来源
- 保留 vendor `CLAUDE.md` 原文不动，继续作为代码任务的原则来源
- 新会话启动由 `~/.codex/BOOTSTRAP.md` 承载压缩版原则，本文件按需读取
- 本文件只负责定义“何时以 Karpathy 为总纲，以及本地规则如何退居执行细则”
- 当本地表述与 vendor 原意不一致时，优先保持 vendor 原意

## 作用域划分

### 1. 代码任务
以下任务默认视为代码任务：
- 写代码
- 改代码
- 调试
- 修 bug
- 重构
- 写测试 / 改测试
- CI / build / typecheck / lint 修复
- 影响软件行为的配置修改
- 代码评审

对这类任务：
- vendor `CLAUDE.md` 是详细行为总纲；启动上下文只保留其压缩版原则
- 本地 `global-rules/dev/*`、`global-rules/delivery/*`、`global-rules/core/*` 默认只作为执行细则
- 执行细则不得稀释或反向推动 vendor 中的四条原则：显式假设、简单优先、手术式改动、目标驱动验证

### 2. 非代码任务
以下任务默认视为非代码任务：
- 文档整理
- 知识库维护
- 归档与清理
- 命名整理
- 内部沟通稿
- 非软件行为的管理类操作

对这类任务：
- 继续以本地 `GLOBAL-AGENT.md` 与 `global-rules/docs/*` 为主
- Karpathy 仅保留通用启发：显式说明假设、优先简单、避免无关改动、结果可验证

### 3. 混合任务
若一个任务同时包含代码与文档/治理工作：
- 代码部分按 Karpathy 总纲执行
- 文档与治理部分按本地规则执行
- 不因为文档要求而放大代码改动范围
- 不因为代码要求而破坏文档结构治理

## 代码任务优先级
对代码任务，按以下顺序理解冲突：
1. 用户当前明确指令
2. `.agent/local/90-project-specific.md`
3. vendor `CLAUDE.md`
4. 本文件 `KARPATHY-INTEGRATION.md`
5. `~/.codex/global-rules/dev/*`
6. `~/.codex/global-rules/delivery/*`
7. `~/.codex/global-rules/core/*`
8. `~/.codex/GLOBAL-AGENT.md`
9. 通用最佳实践

对非代码任务，沿用本地 `03-priority.md` 的默认优先级。

## 本地规则降级方式
对代码任务，本地规则保留这些职责：
- 语言与输出风格
- 项目分析和最小必要上下文读取
- 安全约束
- 验证顺序与结果汇报
- 部署、提交、归档等工程 hygiene

但本地规则不再用于：
- 强推比任务本身更重的设计流程
- 用“先定义结构”推动简单任务过度规划
- 用“长期可维护性”之名引入超前抽象
- 用“顺手整理”扩大代码改动边界
- 因流程完整性而稀释手术式改动

## 流程与技能使用约束
对代码任务：
- 明确、范围清晰、成功标准清楚的小型任务，默认直接进入最小闭环实现与验证
- 只有当歧义会影响正确性、范围或实现方向时，才先澄清
- `brainstorming`、`writing-plans`、spec/plan 审批流主要用于开放式功能设计、架构调整、复杂交互/产品设计，不作为每个小型代码任务的默认前置门槛
- 过程性 skill 是工具，不是为了流程而流程的强制负担

## 生效目标
这套融合后的目标是：
- 尽可能保留原本 Karpathy 的原文与判断风格
- 让代码任务先受 Karpathy 约束，再落到本地工程细则
- 让非代码任务继续由本地中文规则体系稳定承接
- 让全局配置更薄、更清楚、更少重复
